全场景说明（以当前代码为准）

1. 开始 -> 暂停 -> 恢复 -> 结束
- 初始化后点击开始，任务进入播放态。
- 中途点击暂停，收到 PAUSE，播放停止。
- 再次点击开始，收到 START，继续播放。
- 任务可以一直播放到结束，结束时顺序为 PLAY -> PAUSE -> FINISH。

2. 播放中调速为 5 倍 -> 继续播放 -> 结束
- 任务开始播放后，中途将倍速调整为 5。
- 调速时会先收到 SPEED。
- 如果原本就在播放，中间不会进入暂停态，不会收到额外 PAUSE。
- 后续按 5 倍速继续播放，直到正常结束。

3. 初始化后先调 5 倍速 -> 再播放
- 页面初始化完成后，先调 5 倍速。
- 当前代码下，调速为非 0 时会自动触发播放。
- 如果随后再次点击播放，只会收到 START，不会报错，也不会暂停。
- 整个过程可以正常播放到结束。

4. 初始化后先调 5 倍速 -> 播放一段 -> 暂停 -> 恢复 -> 结束
- 初始化完成后，先调 5 倍速并进入播放。
- 播放过程中点击暂停，收到 PAUSE。
- 再次点击开始，收到 START，继续按 5 倍速播放。
- 后续可以正常播放到结束，过程中不应抛出异常。

5. 播放中跳点 -> 继续播放 -> 结束
- 任务开始播放后，中途拖动进度条到指定时间点。
- 后端会收到 skip，请求处理后推送 SKIP。
- 如果原本在播放，跳点后会继续保持播放态。
- 跳点之后可以继续播放，直到正常结束。

6. 播放中暂停 -> 跳点 -> 自动恢复播放 -> 结束
- 任务开始播放后先暂停，收到 PAUSE。
- 暂停状态下拖动进度条到指定时间点。
- 当前代码会自动恢复播放，不需要再次点击开始。
- 常见消息顺序为 START -> SKIP -> PLAY。
- 跳点后任务会继续播放，直到正常结束。

7. 调速和跳点的统一规则
- 调速到非 0 倍速，视为继续播放，应自动进入播放态。
- 拖动进度条跳点，视为继续播放，应自动进入播放态。
- 这两类操作都不应把任务留在暂停态。

8. 播放结束后重新播放
- 任务播放到 FINISH 后，再次点击播放。
- 后端会重置到起点并重新发送 INIT。
- 任务可以从 0 重新开始播放。

9. 全量间隔与播放帧发送规则
- 当前全量间隔单位是秒。
- 如果间隔配置为 10 分钟，则等价于 600 秒。
- 在 0 分钟、10 分钟、20 分钟等整间隔点，播放帧只发送该秒点全量数据。
- 在非整间隔点，播放帧只发送本次步长区间内的增量数据，不发送全量数据。

10. 非整点播放帧示例
- 播放到 1 分钟：只发送第 1 分钟增量数据，不发送全量数据。
- 播放到 2 分钟：只发送第 2 分钟增量数据，不发送全量数据。
- 播放到 11 分钟：只发送 10 到 11 分钟这一分钟的增量数据，不发送全量数据。

11. 跳点发送规则
- 跳点时，系统会发送“距离目标时间最近的全量数据 + 之后到目标点的累计增量数据”。
- 如果目标时间正好落在整间隔点，则只发送该时间点全量数据，不再补增量。

12. 跳点示例（按 10 分钟间隔）
- 跳到 1 分钟：发送 0 分钟全量数据，再加上 0 到 1 分钟的增量数据。
- 跳到 3 分钟：发送 0 分钟全量数据，再加上 0 到 3 分钟的增量数据。
- 跳到 10 分钟：直接发送 10 分钟全量数据。
- 跳到 12 分钟：发送 10 分钟全量数据，再加上 10 到 12 分钟的增量数据。

13. 协议说明
- 以上“全量数据 / 增量数据”描述的是后端快照拼装逻辑。
- 当前对外协议里，前端主消费字段是 `data` 和 `eventData`。
- `fullData` 与 `incrementalData` 当前固定为空数组，仅作兼容保留。

结论
- 当前代码符合以上场景要求。
- 自动播放规则包括两类操作：调速到非 0、跳点。
- 结束后支持重新播放。
- 全量间隔点、非整点播放帧、跳点快照规则与当前代码一致。
