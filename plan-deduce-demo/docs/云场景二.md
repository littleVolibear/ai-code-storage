# 云场景二：先设 3 倍速 -> 首次开始 -> 暂停 -> 继续播放 -> 结束

## 结论

当前代码支持这个场景，验证用例是 `PlanDeduceIntegrationTest.shouldSupportDocumentedScenarioTwoFlow`。

这个场景的关键点是：

1. 先调 `speed=3` 只是预设倍速，不会自动开始播放。
2. 随后第一次调用 `startOrStop?flag=1` 时，如果任务还没初始化，后端会自动初始化。
3. 因为这是首次开始，所以先推 `INIT`，不是 `START`。
4. 后续暂停再恢复时，才会推 `START`。

## 推荐时序

```text
1. 建立 WebSocket
2. GET /plan/speed?dbName=plandeduce&speed=3&sessionId=s2
3. WS: SPEED(currentTime=0, speed=3, running=false)
4. GET /plan/startOrStop?dbName=plandeduce&flag=1&sessionId=s2
5. WS: INIT(currentTime=0, speed=3, running=true)
6. WS: PLAY(currentTime=3)
7. WS: PLAY(currentTime=6)
8. GET /plan/startOrStop?flag=0
9. WS: PAUSE
10. GET /plan/startOrStop?flag=1
11. WS: START
12. WS: PLAY(...)
13. 结束时: PLAY -> PAUSE -> FINISH
```

## 行为说明

### 第一步：预设倍速

请求：

```text
GET /plan/speed?dbName=plandeduce&speed=3&sessionId=s2
```

返回：

- HTTP `200`
- WS `SPEED`

典型状态：

```json
{
  "type": "SPEED",
  "currentTime": 0,
  "speed": 3,
  "running": false
}
```

说明：

- 这时任务可能还没初始化。
- `running=false` 是正常的。

### 第二步：首次开始

请求：

```text
GET /plan/startOrStop?dbName=plandeduce&flag=1&sessionId=s2
```

首次开始的真实行为：

1. 若 `initialized=false`，内部先做初始化。
2. 立刻推 `INIT`。
3. 后续每个 tick 按 `speed=3` 推 `PLAY`。

典型消息：

```json
{
  "type": "INIT",
  "currentTime": 0,
  "speed": 3,
  "running": true
}
```

```json
{
  "type": "PLAY",
  "currentTime": 3,
  "fullTime": 0,
  "speed": 3,
  "running": true
}
```

### 第三步：暂停后恢复

暂停：

```text
GET /plan/startOrStop?dbName=plandeduce&flag=0&sessionId=s2
```

恢复：

```text
GET /plan/startOrStop?dbName=plandeduce&flag=1&sessionId=s2
```

恢复时的真实行为：

- 推 `START`
- 不会再推第二次 `INIT`
- 后续继续按 `speed=3` 推 `PLAY`

## 与现有协议一致的点

1. `sendPlanDeduce` 不是这个场景的必需步骤。
2. 快照类消息里前端消费 `data` 和 `eventData`。
3. `fullData`、`incrementalData` 当前固定为空数组。
4. 进度跳变要按 `PLAY.currentTime` 看，3 倍速时会是 `0 -> 3 -> 6 -> 9 ...`。
