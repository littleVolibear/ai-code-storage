# 云场景二：先设 3 倍速 -> 自动开始 -> 暂停 -> 继续播放

## 结论

当前代码支持这个场景，对应测试是 `PlanDeduceIntegrationTest.shouldSupportDocumentedScenarioTwoFlow`。

以现在的代码为准，这个场景最重要的结论不是“先设倍速只预设不启动”，而是：

1. 调 `/plan/speed?speed=3` 会先推 `SPEED`。
2. 只要此时任务不在运行，并且 `speed > 0`，当前代码会自动调用 `resume()`。
3. 如果任务还没初始化，这次 `speed` 请求后就会进入 `INIT`，随后开始 `PLAY`。
4. 之后再暂停和恢复，恢复时才会收到 `START`。

也就是说，当前真实行为是：

- 未初始化时先调 `speed=3`，消息顺序是 `SPEED -> INIT -> PLAY`
- 不是“只收 `SPEED`，还要再手动调一次开始接口才会 `INIT`”

## 推荐时序

```text
1. 建立 WebSocket
2. GET /plan/speed?dbName=plandeduce&speed=3&sessionId=s2
3. WS: SPEED(realTime=0, deduceTime=0, speed=3, running=false)
4. WS: INIT(realTime=0, deduceTime=0, speed=3, running=true)
5. WS: PLAY(realTime=1, deduceTime=3)
6. WS: PLAY(realTime=2, deduceTime=6)
7. GET /plan/startOrStop?dbName=plandeduce&flag=0&sessionId=s2
8. WS: PAUSE(realTime=2, deduceTime=6, speed=3, running=false)
9. GET /plan/startOrStop?dbName=plandeduce&flag=1&sessionId=s2
10. WS: START(realTime=2, deduceTime=6, speed=3, running=true)
11. WS: PLAY(realTime=3, deduceTime=9)
```

## 行为说明

### 第一步：设置倍速

请求：

```text
GET /plan/speed?dbName=plandeduce&speed=3&sessionId=s2
```

真实行为：

- 先推 `SPEED`
- 因为 `speed > 0` 且当前 `running=false`，会自动恢复播放
- 如果还没初始化，会继续推 `INIT`

典型状态：

```json
{
  "type": "SPEED",
  "realTime": 0,
  "deduceTime": 0,
  "speed": 3,
  "running": false
}
```

```json
{
  "type": "INIT",
  "realTime": 0,
  "deduceTime": 0,
  "speed": 3,
  "running": true
}
```

### 第二步：播放推进

3 倍速下：

- `realTime` 仍然按 tick 一次加 `1`
- `deduceTime` 按业务时间一次加 `3`

所以典型推进会是：

- `PLAY(realTime=1, deduceTime=3)`
- `PLAY(realTime=2, deduceTime=6)`
- `PLAY(realTime=3, deduceTime=9)`

### 第三步：暂停后恢复

暂停：

```text
GET /plan/startOrStop?dbName=plandeduce&flag=0&sessionId=s2
```

恢复：

```text
GET /plan/startOrStop?dbName=plandeduce&flag=1&sessionId=s2
```

恢复时的真实行为：

- 推 `START`
- 不会再次推 `INIT`
- 后续继续按 `speed=3` 推 `PLAY`

## 协议注意点

1. 这个场景里，`speed=3` 已经足以自动拉起播放。
2. 前端时间显示要用 `deduceTime`，不要把 `realTime` 当成业务时间。
3. 当前快照主消费字段是 `data` 和 `eventData`。
4. `fullData` / `incrementalData` 对外固定为空数组，仅做兼容保留。
