# 传输方案比较

本文档用于说明当前项目为什么采用 WebSocket，以及如果改成纯 HTTP 接口交互是否可行，并对两种方式做比较。

## 当前项目为什么适合 WebSocket

这个项目的后端任务不是“前端请求一次，后端返回一次”这么简单，而是：

- 任务启动后会持续推进
- 时间会自动变化
- 后端会持续产生新的状态和数据
- 前端需要实时刷新进度条、播放状态和数据快照

这类场景天然适合 WebSocket，因为它允许后端主动推送消息给前端。

## WebSocket 方案

### 适用方式

- HTTP：前端发控制命令
- WebSocket：后端主动推送状态和数据

### 优势

1. 实时性好  
后端一有新状态就可以立刻推给前端，不需要前端不断轮询。

2. 请求开销更低  
连接建立一次后可以持续收消息，不需要频繁发 HTTP 请求。

3. 更适合表达事件流  
像 `INIT`、`PLAY`、`PAUSE`、`START`、`SPEED`、`SKIP`、`FINISH`、`DESTROY`、`ERROR` 这种消息，本质上就是事件流，WebSocket 很适合承载这种模型。

4. 前后端职责清晰  
- HTTP：发控制指令
- WebSocket：收状态和数据

### 弊端

1. 实现复杂度更高  
需要处理连接建立、断线重连、session 路由、多页面并发等问题。

2. 调试和排障更复杂  
相比普通接口，请求链路和复现方式都更麻烦。

3. 状态一致性要求更高  
前端必须以消息推送为准，不能只根据按钮点击更新 UI。

4. 长连接资源管理要求更高  
连接数增加后，需要考虑网关、内存、线程和连接管理策略。

## 纯 HTTP 接口交互方案

如果不用 WebSocket，也可以做成纯 HTTP 模式，但通常会演化成以下形式：

### 方案一：轮询

前端定时调用一个查询接口，例如：

- `GET /plan/status`
- `GET /plan/currentData`

比如每 `500ms` 或 `1s` 调用一次。

### 方案二：长轮询 / SSE

仍然属于 HTTP 体系，但比普通轮询更接近推送。

## 纯 HTTP 方案的优势

1. 实现和接入更简单  
大多数前后端都更熟悉普通 HTTP 模式。

2. 调试更方便  
接口请求和响应都更容易抓包、记录、复现。

3. 基础设施兼容性更好  
普通 HTTP 在代理、网关、日志、监控方面支持更成熟。

## 纯 HTTP 方案的弊端

1. 实时性通常不如 WebSocket  
前端只能定时去问，而不是后端即时推送。

2. 会产生更多无效请求  
即使状态没变化，前端仍要重复拉取。

3. 状态延迟更明显  
比如用户点了暂停，前端可能要等下一次轮询才知道真正的状态。

4. 事件语义不够直观  
像 `PAUSE`、`START`、`FINISH` 这类事件，在纯轮询模型里通常只能通过字段变化间接判断，不如直接推送明确。

## 如果改成纯 HTTP，这个项目大概会怎么设计

可以拆成以下接口：

- `POST /plan/start`
- `POST /plan/pause`
- `POST /plan/speed`
- `POST /plan/skip`
- `POST /plan/destroy`
- `GET /plan/status?dbName=...&sessionId=...`

前端逻辑大致变成：

1. 用户点开始，调 `/start`
2. 前端每隔一段时间调 `/status`
3. 后端返回当前状态，例如：

```json
{
  "running": true,
  "currentTime": 15,
  "fullTime": 10,
  "speed": 5,
  "data": [],
  "finished": false
}
```

这种方式完全可行，只是它是“拉模型”，不是“推模型”。

## SSE 作为折中方案

如果前端只需要后端单向推送，而不需要在同一个连接上双向通信，那么 SSE 也是一个折中方案。

### SSE 的优点

- 比 WebSocket 简单
- 比轮询更实时
- 仍然是 HTTP 体系，调试通常比 WebSocket 更容易

### SSE 的限制

- 主要适合服务端单向推送
- 浏览器和网关支持通常没问题，但灵活性不如 WebSocket

## 三种方案对比

### WebSocket

- 实时性：高
- 实现复杂度：高
- 调试成本：中到高
- 适合：播放型、推演型、实时状态型系统

### HTTP 轮询

- 实时性：一般
- 实现复杂度：低
- 调试成本：低
- 适合：实时要求不高、优先简单稳定的系统

### SSE

- 实时性：较高
- 实现复杂度：中
- 调试成本：中
- 适合：后端单向推送状态、但不想引入完整 WebSocket 复杂度的系统

## 对当前项目的建议

从当前项目的业务模型看：

- 进度条需要持续推进
- 后端会不断产生新的播放状态
- 前端需要及时接收 `INIT/PLAY/PAUSE/START/SPEED/SKIP/FINISH/DESTROY/ERROR`

因此，当前采用 WebSocket 是合理的，而且是比较匹配的方案。

如果你的优先级变成：

- 更容易让普通前端同学接入
- 更简单、更稳定、更好排查问题
- 对实时性要求没那么高

那就可以考虑改成：

- HTTP 控制 + HTTP 轮询状态

如果想在“实时性”和“复杂度”之间折中，可以考虑：

- HTTP 控制 + SSE 推送状态

## 结论

对于当前这个进度条播放/推演项目：

- WebSocket：合理，适合实时播放型场景
- 纯 HTTP：可行，但通常会退化成轮询模型，实时性和事件表达较弱
- SSE：可以作为中间方案

最终选型应根据你的核心目标来决定：

- 重视实时性和事件驱动：优先 WebSocket
- 重视简单和易维护：优先 HTTP 轮询
- 想折中：考虑 SSE
