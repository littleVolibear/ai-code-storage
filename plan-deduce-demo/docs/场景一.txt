这个项目的实际交互模型很明确：HTTP 只负责“发控制命令”，WebSocket 才是前端唯一可信的状态源。控制器注释和实现就是这么设计的，见 PlanDeduceController.java；而 WebSocket 未连接时，后端推送会直接丢弃，见 PlanDeduceWebSocketHandler.java。

针对“点击播放 -> 暂停 -> 再播放 -> 最后结束”这个场景，建议前端按下面的方式调后端。

建议流程
1. 页面进入先建立 ws://.../ws/planDeduce?sessionId=xxx。
2. 用户第一次点播放时，调用 GET /plan/sendPlanDeduce?dbName=...&skip=0&sessionId=xxx。
3. 前端等 INIT 消息后完成初始化，再等后续 PLAY 推进进度。
4. 用户点暂停时，调用 GET /plan/startOrStop?dbName=...&flag=0&sessionId=xxx。
5. 前端收到 PAUSE 后，再把 UI 切成暂停态。
6. 用户点继续播放时，调用 GET /plan/startOrStop?dbName=...&flag=1&sessionId=xxx。
7. 前端收到 START 后标记恢复中，真正进度推进仍以后续 PLAY 为准。
8. 播到终点时，后端会先发最后一帧 PLAY，再发 PAUSE，最后发 FINISH，这一点在 ScenarioTask.java 和 PlanDeduceIntegrationTest.java 里都能看到。

前端职责
- 要保存 dbName、sessionId、initialized、running、finished、currentTime、fullTime、speed、fullData、incrementalData、data。
- 但这些状态的更新必须以后端推送为准，不要因为 HTTP 200 就直接改 UI。
- 不要自己本地计时推进进度条，currentTime 只认 PLAY.currentTime。
- 同一个页面实例内 sessionId 必须固定；多标签页不要复用。
- 建议前端对控制命令做串行化处理：一个命令发出后，等对应的 WS 事件到了再允许下一个关键操作。因为当前协议里没有 requestId，只能靠时序收敛。

几个关键建议
- sendPlanDeduce 是“初始化并开始播放”，不是“普通恢复播放”。恢复播放应该走 startOrStop(flag=1)，见 PlanDeduceController.java。
- 非初始化命令都用了 getOrCreate()，但真正拉起 worker 只在 start() 里做，见 PlanDeduceController.java 和 ScenarioTask.java。所以前端不要在未初始化时调用 skip、speed、startOrStop、fullSaveInterval，否则可能创建出“半初始化任务”。
- 暂停建议优先用 startOrStop(flag=0)，不要优先用 speed=0。因为 speed=0 之后恢复播放时，后端会把速度兜底成 1，不会自动恢复到之前的 3 或 5，见 ScenarioTask.java。
- 如果产品一定要把“倍速调成 0”当暂停，那前端就必须自己记住“上一次非 0 倍速”，恢复前先补一次 /plan/speed?speed=原倍速，再调 /plan/startOrStop?flag=1。

下面补充“场景一”里前端发起请求后，后端具体会回什么内容。

一、HTTP 响应
这个项目里的这几个控制接口，HTTP 层基本都只返回状态码，不返回业务数据：

1. GET /plan/sendPlanDeduce
2. GET /plan/startOrStop
3. GET /plan/skip
4. GET /plan/speed
5. GET /plan/fullSaveInterval
6. GET /plan/destroy

正常情况下：
- HTTP 状态码是 200
- 响应体为空

异常情况下：
- 例如重复初始化、非法参数
- HTTP 状态码通常是 400
- 响应体格式大致如下：

{
  "success": false,
  "message": "错误说明"
}

所以前端不能指望 HTTP 响应体拿到播放状态、当前时间或数据快照，真正的数据要等 WebSocket。

二、WebSocket 消息总格式
后端 WebSocket 发回的是 JSON，对应 PushMessage。整体格式如下：

{
  "type": "INIT | PLAY | PAUSE | START | FINISH | SKIP | SPEED | INTERVAL | DESTROY | ERROR",
  "dbName": "plandeduce",
  "sessionId": "s1",
  "currentTime": 0,
  "fullTime": 0,
  "speed": 1,
  "running": true,
  "fullData": [],
  "incrementalData": [],
  "data": [],
  "message": "说明文案"
}

字段说明：
- type：消息类型
- dbName：当前任务所属数据库
- sessionId：当前会话标识
- currentTime：当前播放到的业务秒点
- fullTime：当前快照对应的最近全量秒点
- speed：当前倍速
- running：当前是否处于播放态
- fullData：全量快照数据，只在快照类消息里有值
- incrementalData：增量数据，只在快照类消息里有值
- data：后端直接合并好的结果，等于 fullData + incrementalData
- message：说明文案，方便日志或联调查看

注意：
- INIT、PLAY、SKIP、INTERVAL 这类“快照类消息”通常带 fullData、incrementalData、data
- PAUSE、START、FINISH、SPEED、DESTROY、ERROR 这类“状态类消息”通常只有基础状态字段和 message，不带数据数组

2. 一条 PAUSE
3. 一条 FINISH

随后 PAUSE：
{
  "type": "PAUSE",

说明：
- 前端不要只靠 currentTime >= fullTime 或 currentTime 到最大值来猜测结束
- 必须以 FINISH 为准，把 finished 置为 true
- 因为结束前会先收到一条 PAUSE，所以前端状态机会更稳妥的写法是：
  收到 PAUSE 时只表示停止推进
  收到 FINISH 时再表示本次任务真正结束

四、数据数组里的对象格式
fullData、incrementalData、data 三个数组里的元素，都是 RoomObjectHis 对象。字段很多，前端通常只需要消费一部分核心字段。常见格式如下：

{
  "roomObjectId": 1,
  "scenarioId": 1001,
  "roomId": 501,
  "objCode": "RED_ROOK",
  "side": "RED",
  "objName": "Red Rook Battalion",
  "currentPos": 100,
  "nextPos": 101,
  "direction": 0,
  "currentSpeed": 800,
  "moving": 0,
  "stopping": 0,
  "visible": 1,
  "simTime": 0,
  "targetId": 1,
  "sourceType": "FULL"
}

补充说明：
- sourceType 是后端补充的来源标识，通常是 FULL 或 INCREMENT
- data 是后端直接拼好的 fullData + incrementalData
- 如果前端只想直接渲染当前结果，可以优先消费 data
- 如果前端自己要做更细的快照/增量逻辑，也可以分别消费 fullData 和 incrementalData

五、前端在场景一里应该如何使用这些返回
- 第一次收到 INIT：建立初始页面状态
- 每次收到 PLAY：推进 currentTime，并刷新页面数据
- 收到 PAUSE：只切到暂停态，不代表结束
- 收到 START：只切到播放态，真正进度以后续 PLAY 为准
- 收到 FINISH：设置 finished=true，禁用继续播放按钮，或者引导用户重新初始化

六、这个场景里最推荐的前端判断规则
- HTTP 200 只表示“命令提交成功”，不表示 UI 已完成切换
- INIT、PLAY、SKIP、INTERVAL 负责数据刷新
- START、PAUSE、SPEED、FINISH、DESTROY、ERROR 负责状态切换
- 结束判断只认 FINISH
- 当前是否在播放只认 running
- 当前显示到哪一秒只认 currentTime
