# 云场景一：初始化 -> 开始播放 -> 暂停 -> 继续播放 -> 结束

## 结论

当前代码支持这个场景，验证用例是 `PlanDeduceIntegrationTest.shouldStartPauseAndResumeAtOneX`。

这份文档只描述当前真实行为：

1. `sendPlanDeduce` 只初始化任务，返回进度条时间范围，不推 WebSocket 数据。
2. 第一次真正调用 `startOrStop?flag=1` 时，后端先推 `INIT`，再开始持续推 `PLAY`。
3. 暂停时推 `PAUSE`。
4. 暂停后继续播放时推 `START`，随后继续推 `PLAY`。
5. 到终点时顺序是最后一帧 `PLAY` -> `PAUSE` -> `FINISH`。

## 推荐时序

```text
1. 建立 WebSocket: /ws/planDeduce?sessionId=s1
2. GET /plan/sendPlanDeduce?dbName=plandeduce&skip=0&sessionId=s1
3. HTTP 返回 InitProgressResponse:
   {
     "dbName": "plandeduce",
     "sessionId": "s1",
     "startTime": "2026-01-01 00:00:00",
     "endTime": 1200
   }
4. GET /plan/startOrStop?dbName=plandeduce&flag=1&sessionId=s1
5. WS: INIT
6. WS: PLAY(currentTime=1)
7. WS: PLAY(currentTime=2)
8. GET /plan/startOrStop?dbName=plandeduce&flag=0&sessionId=s1
9. WS: PAUSE
10. GET /plan/startOrStop?dbName=plandeduce&flag=1&sessionId=s1
11. WS: START
12. WS: PLAY(...)
13. WS: PLAY(currentTime=maxSimTime)
14. WS: PAUSE
15. WS: FINISH
```

## HTTP 侧约定

`/plan/sendPlanDeduce`

- 作用：只初始化任务和返回进度条范围。
- 不会推 `INIT`。
- 如果同一个 `dbName + sessionId` 任务正在执行，会返回 `400`。

`/plan/startOrStop?flag=1`

- 第一次开始：推 `INIT`，不是 `START`。
- 已初始化后恢复播放：推 `START`。

`/plan/startOrStop?flag=0`

- 推 `PAUSE`。

## WebSocket 消息约定

快照类消息：`INIT`、`PLAY`、`SKIP`、`INTERVAL`

- 前端实际应消费 `data`。
- `fullData` 和 `incrementalData` 当前固定为空数组，只为兼容字段保留。
- `eventData` 会随帧一起返回 `FireJudgeResult` 数据。

状态类消息：`PAUSE`、`START`、`SPEED`、`FINISH`、`DESTROY`、`ERROR`

- 不带 `data`。
- 以前端收到的 WS 消息作为最终状态来源，不要只看 HTTP 200。

## 当前 PushMessage 关键字段

```json
{
  "type": "INIT | PLAY | SKIP | INTERVAL | PAUSE | START | SPEED | FINISH | DESTROY | ERROR",
  "dbName": "plandeduce",
  "sessionId": "s1",
  "currentTime": 0,
  "fullTime": 0,
  "speed": 1,
  "running": true,
  "maxSimTime": 1200,
  "fullData": [],
  "incrementalData": [],
  "data": [],
  "eventData": [],
  "message": "..."
}
```

## 前端注意事项

1. 页面进入先建 WebSocket，再发 HTTP 控制命令。
2. 进度条开始时间、结束时间来自 `sendPlanDeduce` 返回体，不来自 WS。
3. 地图和进度只以后端推送的 `currentTime` 为准，不要本地自增。
4. 恢复播放要等 `START`，真正推进要等后续 `PLAY`。
5. 同一个页面实例内 `sessionId` 要固定，多标签页不要复用。
