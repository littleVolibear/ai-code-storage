# 云场景一：初始化 -> 开始播放 -> 暂停 -> 继续播放 -> 结束

## 结论

当前代码支持这个场景，对应测试是 `PlanDeduceIntegrationTest.shouldStartPauseAndResumeAtOneX`。

以现在的代码为准，这个场景的真实行为是：

1. `sendPlanDeduce` 只初始化任务并返回时间范围，不推 WebSocket。
2. 第一次 `startOrStop?flag=1` 会先推 `INIT`，随后持续推 `PLAY`。
3. 暂停时推 `PAUSE`。
4. 暂停后再次 `startOrStop?flag=1` 时推 `START`，随后继续推 `PLAY`。
5. 到终点时顺序是最后一帧 `PLAY` -> `PAUSE` -> `FINISH`。

## 推荐时序

```text
1. 建立 WebSocket: /ws/planDeduce?sessionId=s1
2. GET /plan/sendPlanDeduce?dbName=1&skip=0&sessionId=s1
3. HTTP 返回:
   {
     "dbName": "1",
     "sessionId": "s1",
     "startTime": "2026-01-01 00:00:00",
     "endTime": 1200
   }
4. GET /plan/startOrStop?dbName=1&flag=1&sessionId=s1
5. WS: INIT(realTime=0, deduceTime=0)
6. WS: PLAY(realTime=1, deduceTime=1)
7. WS: PLAY(realTime=2, deduceTime=2)
8. GET /plan/startOrStop?dbName=1&flag=0&sessionId=s1
9. WS: PAUSE(realTime=2, deduceTime=2)
10. GET /plan/startOrStop?dbName=1&flag=1&sessionId=s1
11. WS: START(realTime=2, deduceTime=2)
12. WS: PLAY(...)
13. WS: PLAY(realTime=1199, deduceTime=1200)
14. WS: PAUSE(realTime=1199, deduceTime=1200)
15. WS: FINISH(realTime=1199, deduceTime=1200)
```

## HTTP 侧约定

`/plan/sendPlanDeduce`

- 只初始化任务和返回进度条范围。
- 不会主动推 `INIT`。
- 如果同一个 `sessionId` 的任务正在执行，会返回 `400`。

`/plan/startOrStop?flag=1`

- 首次开始：推 `INIT`。
- 暂停后恢复：推 `START`。

`/plan/startOrStop?flag=0`

- 推 `PAUSE`。

## WebSocket 消息约定

快照类消息：`INIT`、`PLAY`、`SKIP`、`INTERVAL`

- 前端主消费字段是 `data` 和 `eventData`。
- `fullData` / `incrementalData` 当前仍保留在协议里，但对外固定为空数组。
- 当前时间字段分成两条轴：
  - `realTime`：真实 tick 时间
  - `deduceTime`：推演业务时间

状态类消息：`PAUSE`、`START`、`SPEED`、`FINISH`、`DESTROY`、`ERROR`

- 不带快照数据。
- 前端应以后端 WS 消息作为最终状态来源，不要只看 HTTP 200。


## 前端注意事项

1. 页面进入先建 WebSocket，再发 HTTP 控制命令。
2. 进度条开始时间、结束时间来自 `sendPlanDeduce` 返回体，不来自 WS。
3. 地图和进度只以后端推送的 `deduceTime` 为准，不要本地自增。
4. 恢复播放要等 `START`，真正推进要等后续 `PLAY`。
5. 当前任务唯一维度是 `sessionId`，同一个页面实例内不要复用给多个任务。
