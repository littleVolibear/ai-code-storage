# 云场景三：1 倍速播到 5 秒 -> 暂停 -> 改 3 倍速 -> 继续播放 -> 结束

## 结论

当前代码支持这个场景，验证用例是 `PlanDeduceIntegrationTest.shouldSupportDocumentedScenarioThreeFlow`。

这个场景的关键点是：

1. 先通过 `sendPlanDeduce` 初始化。
2. 再通过第一次 `startOrStop?flag=1` 开始播放。
3. 播到 `currentTime >= 5` 后暂停。
4. 暂停状态下改成 3 倍速不会自动恢复播放。
5. 还需要再次调用 `startOrStop?flag=1`，这次收到的是 `START`，然后下一帧会从 `5 -> 8`。

## 推荐时序

```text
1. 建立 WebSocket
2. GET /plan/sendPlanDeduce?dbName=plandeduce&skip=0&sessionId=s3
3. HTTP 返回 InitProgressResponse
4. GET /plan/startOrStop?dbName=plandeduce&flag=1&sessionId=s3
5. WS: INIT(currentTime=0, speed=1)
6. WS: PLAY(currentTime=1)
7. WS: PLAY(currentTime=2)
8. WS: PLAY(currentTime=3)
9. WS: PLAY(currentTime=4)
10. WS: PLAY(currentTime=5)
11. GET /plan/startOrStop?flag=0
12. WS: PAUSE(currentTime=5, speed=1, running=false)
13. GET /plan/speed?speed=3
14. WS: SPEED(currentTime=5, speed=3, running=false)
15. GET /plan/startOrStop?flag=1
16. WS: START(currentTime=5, speed=3, running=true)
17. WS: PLAY(currentTime=8)
18. 后续 11, 14, 17 ...
19. 结束时: PLAY -> PAUSE -> FINISH
```

## 前端判断暂停时机

正确做法：

- 以前端收到的 `PLAY.currentTime >= 5` 作为“已经播放到 5 秒”的依据。

不要这样做：

- 本地自己计时 5 秒后直接认为可以暂停。

原因：

- 当前播放推进完全以后端 `PLAY` 消息为准。
- 测试环境里 tick 可能是 100ms，生产配置也可能调整，不能把“真实业务秒点”理解成“浏览器本地 5 秒”。

## 关键消息语义

### 初始化

`sendPlanDeduce` 只返回：

```json
{
  "dbName": "plandeduce",
  "sessionId": "s3",
  "startTime": "2026-01-01 00:00:00",
  "endTime": 1200
}
```

不会直接推 `INIT`。

### 第一次开始

第一次 `startOrStop?flag=1` 推 `INIT`，随后推 `PLAY`。

### 暂停后调速

`speed=3` 只会更新速度：

```json
{
  "type": "SPEED",
  "currentTime": 5,
  "speed": 3,
  "running": false
}
```

不会自动继续播放。

### 再次开始

第二次 `startOrStop?flag=1` 推 `START`：

```json
{
  "type": "START",
  "currentTime": 5,
  "speed": 3,
  "running": true
}
```

下一帧 `PLAY.currentTime=8`，这就是 3 倍速恢复后的真实推进。

## 数据字段约定

1. 地图渲染使用 `data`。
2. 事件渲染使用 `eventData`。
3. `fullData`、`incrementalData` 当前固定为空数组，不做业务消费。
4. 如果 `currentTime` 命中全量间隔点，`PLAY.fullTime` 会跳到该间隔点；否则保持最近全量点。
